## Discussion Notes

### Key Decisions

| Decision | Reasoning |
|---|---|
| Short description added to cabin profile | Host needs to describe the cabin beyond name/location |
| Amenity tags are utility-focused (heater, AC, water storage) | Reflects actual cabin needs vs. lifestyle amenities |
| Amenity tags set during the create call, not a separate step | Reduces round trips; Host can set them at registration time |
| Cabin name max 512 characters | Prevents abuse while allowing descriptive names |
| Cabin name unique per Host | Prevents confusion in the Host's own dashboard |
| Description uncapped | No practical reason to limit for MVP |
| Update Cabin Profile uses PUT semantics | Simpler to reason about; all fields required on update |
| Amenity tag update replaces the full set | Consistent with create; no partial tag merge |
| Optimistic concurrency via `version` field | Prevents silent data loss on concurrent edits |
| List includes inactive cabins with `is_active: false` | Host needs visibility of all their cabins |
| List ordered by `created_at` DESC | Most recently added cabin appears first |
| Key info in separate table, masked at API layer | Keeps sensitive fields away from public profile; masking in service layer not DB |
| Reveal audit log on `reveal=true` | Security requirement; records cabin ID, host ID, timestamp |
| Encryption at rest deferred | RLS is the MVP boundary; flagged as future improvement |
| List Amenity Tags is public (no auth required) | Tag list contains no sensitive data |
| Caching on amenity tags deferred | Not needed at MVP scale |

### Deferred / Out of Scope

| Item | Why Deferred |
|---|---|
| Deactivate Cabin | Has cascading effects on bookings; deferred to a later intent |
| Encryption at rest for key info | Complexity not justified for MVP; RLS + masking sufficient |
| Caching on amenity tag list | Not needed at MVP scale |
