## Discussion Notes

### Key Decisions

| Decision | Reasoning |
|---|---|
| Key information stored in a separate `cabin_key_info` table, not on `cabins` | Keeps sensitive fields separate from the public profile; RLS can be scoped more tightly |
| Amenities stored as a join table `cabin_amenity_tags` with a `amenity_tags` lookup table | Supports the predefined tag list; easy to add/remove tags without schema changes |
| `reveal=true` query param returns plaintext key info | Simple toggle; no separate endpoint needed; request is still authenticated + ownership-checked |
| List endpoint returns public profile only — no key info | Prevents bulk exposure of sensitive fields; Host must fetch individual cabin detail to see key info |
| No soft-delete unit in this session | Deactivating a cabin has booking implications — deferred to a booking-adjacent intent |

### Deferred / Out of Scope

| Item | Why Deferred |
|---|---|
| Photo/image upload | Requires storage integration; explicitly excluded from MVP |
| Amenity tag administration (add/remove from predefined list) | Host-facing only; tag management is a separate Host-admin concern |
| Cabin deactivation | Has cascading effects on active bookings; belongs to a booking intent |
| Public cabin listing (Guest view) | Belongs to a booking/discovery intent |

We are building CabinConnect — a cabin booking platform with a .NET Web API backend,
React + TypeScript frontend, and Supabase (PostgreSQL) as the database.

Feature: Cabin Profile Management

capacity, amenities from a predefined tag list) and sensitive key information
(access codes, emergency contacts, house rules). Key info is masked by default
and revealed on explicit request. A Host cannot see or modify another Host's cabin.

For each Unit: name, purpose, Given/When/Then ACs, dependencies.
